Below is a solid **Sway + custom shell v0** plan that gets you to a usable ‚Äúdesktop‚Äù without writing a compositor.

## v0 scope definition

**Goal:** ‚ÄúBoot to a session that feels coherent‚Äù: panel + launcher + notifications + basic settings + sane defaults.
**Non-goals:** custom compositor features, complex animations, full control center, full theming engine.

---

## v0 required features (minimum bar for ‚Äúusable‚Äù)

### Session + lifecycle

* Launch from a display manager (or start from TTY) into a single command: `your-shell-session`
* Clean startup/shutdown of shell components (no orphan daemons)
* Logging (journal or files) and a ‚Äúsafe mode‚Äù fallback (e.g., start Sway with no shell UI)

### Core UX

* **Panel/bar** with:

  * workspace indicator
  * focused window title (optional but helpful)
  * clock
  * tray/status area: network, audio, battery (at least placeholders)
* **App launcher** (search + run `.desktop` apps)
* **Notifications** (implement `org.freedesktop.Notifications` or wrap an existing daemon temporarily)
* **Screenshot + screen recording integration** (at least screenshot; recording can be deferred)
* **Basic power controls**: logout, reboot, shutdown (through `systemctl`/logind)

### Input + keybindings

* Keybinding map (at least: launcher, close window, move focus, switch workspace, screenshot, volume/brightness)
* Media keys working (volume/brightness) using common tools (`pactl/wpctl`, `brightnessctl`)
* One place to define keybindings (even if it writes out Sway config)

### Config and persistence

* Config file format (TOML/YAML/JSON) with versioning/migrations (even minimal)
* Store settings in `~/.config/your-shell/`
* Provide ‚Äúreset to defaults‚Äù

## v0 task list (ordered for fastest path to a working desktop)

### 0) Repo + packaging foundation

* [ ] Pick project name + XDG paths (`~/.config`, `~/.local/share`)
* [ ] Create packages:

  * `your-shell-session` (entry script + `.desktop` for display managers)
  * `your-shell-ui` (panel/launcher)
  * `your-shell-notifd` (optional separate service)
  * `your-shell-settingsd` (optional, can start as config file only)
* [ ] Provide a dev-run script: `./run-in-nested-sway` (launches a nested Sway session for testing)

### 1) Sway session wrapper (make it reproducible)

* [ ] Write a session launcher that:

  * sets env vars
  * starts Sway with a known config
  * starts your shell services
* [ ] Provide a default Sway config template you ‚Äúown‚Äù:

  * inputs, outputs, gaps/borders defaults
  * keybinds that call your shell
  * exec lines for shell components
* [ ] Add fallback: if shell fails, still land in Sway with a simple bar disabled

**Practical defaults to set early**

* `XDG_CURRENT_DESKTOP`, `XDG_SESSION_TYPE=wayland`
* `MOZ_ENABLE_WAYLAND=1`, `QT_QPA_PLATFORM=wayland;xcb`
* `GTK_USE_PORTAL=1`
* (optionally) `SDL_VIDEODRIVER=wayland`

### 2) Panel v0 (start dumb, get pixels on screen)

* [ ] Create a panel window using layer-shell (top layer)
* [ ] Implement: workspaces list (read via `swaymsg -t get_workspaces`)
* [ ] Implement: focused window title (via `swaymsg -t subscribe` events)
* [ ] Implement: clock
* [ ] Add system status placeholders (audio/battery/network) that refresh on timer

### 3) Launcher v0

* [ ] Parse `.desktop` entries (XDG applications dirs)
* [ ] Fuzzy search + run
* [ ] Hotkey (Super/Alt+Space) from Sway config triggers launcher toggle
* [ ] Record MRU (most recently used) to improve results

### 4) Notifications v0

* [ ] Implement `org.freedesktop.Notifications` (D-Bus)
* [ ] Render notification popups via layer-shell (top-right, stacked)
* [ ] Actions: close, default action
* [ ] Persistence not required in v0 (but at least keep visible until timeout)

*(Shortcut option: use `mako` initially, and just style your shell around it; swap later.)*

### 5) Basic settings v0 (no control center yet)

* [ ] Config file with:

  * wallpaper path
  * panel position/size
  * default terminal
  * launcher hotkey
* [ ] CLI tool: `your-shellctl reload`, `your-shellctl set wallpaper ...`
* [ ] Implement ‚Äúreload config‚Äù without restarting the whole session

### 6) Integrations that prevent ‚Äúwhy doesn‚Äôt this work‚Äù bug reports

* [ ] Polkit agent (start one or ship your own minimal agent UI)
* [ ] XDG Desktop Portal situation:

  * ensure an appropriate portal backend is present for wlroots (`xdg-desktop-portal-wlr`) OR document requirement
* [ ] Audio control:

  * support PipeWire/WirePlumber (use `wpctl` when available, fallback to `pactl`)
* [ ] Clipboard manager (optional v0, but have at least sane clipboard behavior)

### 7) Power/session controls

* [ ] Lock screen strategy (even if you just integrate `swaylock` in v0)
* [ ] Idle behavior (integrate `swayidle` in v0)
* [ ] UI menu: logout/reboot/shutdown

### 8) QA + release hygiene

* [ ] Test matrix:

  * multi-monitor hotplug
  * laptop lid close/suspend
  * HiDPI scaling
  * NVIDIA (at least document known issues)
* [ ] Crash handling: if panel dies, restart it (simple supervisor or systemd user units)
* [ ] Document dependencies and how to debug (logs + `swaymsg` tips)

## Important considerations (things that bite early)

### IPC strategy (don‚Äôt paint yourself into a corner)

For v0, it‚Äôs fine to use:

* `swaymsg -t subscribe` for events
* `swaymsg -t get_tree/get_workspaces` for state

But implement a small internal abstraction so you can later swap to:

* direct i3ipc library bindings
* or a dedicated ‚Äúwm-bridge‚Äù service that normalizes events/state for your shell

### Systemd user units vs ‚Äúone big script‚Äù

Strongly consider **systemd user services** for:

* panel
* notifications
* settings daemon

Benefits: auto-restart, logs, dependency ordering.

### Layer-shell and z-ordering

Your shell UI should use **layer-shell** correctly:

* panel: `layer=top`, `exclusive_zone` set so windows don‚Äôt overlap
* notifications: `layer=top`, no exclusive zone
* launcher: `layer=overlay` (typically)

### Portals and sandboxed apps

Without portals, Flatpak apps will have broken:

* file picker
* screenshots/screencast
* open-uri
  Make it a documented requirement at minimum for v0.

### Input + keybinding ownership

Avoid splitting ‚Äútruth‚Äù across:

* Sway config
* your shell config
  Pick one as source of truth (often: your shell config generates/updates a Sway include file).

### Theming: keep it minimal early

Do **not** build a theme engine in v0.
Ship:

* light/dark toggle (optional)
* a few constants for spacing/font sizes
* let users override via a CSS file if using GTK/web UI

### Security posture (since you‚Äôre building a shell)

* Treat any plugin/extension ideas as **out of scope** for v0 unless you have a trust model.
* Be careful with ‚Äúrun command‚Äù features; don‚Äôt accidentally create a privileged command execution path.
