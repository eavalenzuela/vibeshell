Internal state model (Rust-friendly)

Think of it as a small scene graph + constraints, with Sway as the executor.

Core entities

viewport: Viewport (center x,y, scale, per-output optional)

active_cluster: ClusterId (derived from zoom, but useful)

outputs: Vec<OutputState>

Cluster

id: ClusterId

name: String

pos: Vec2 (canvas coordinates)

radius: f32 (for hit-testing / overview layout)

rules: ClusterRules (optional v1: default terminal dir, etc.)

last_focus: Option<WindowId>

Window

id: WindowId (Sway container id)

app_id/class/title

pid: Option<u32> (if you can get it)

role: WindowRole (Normal | Dialog | Utility | Overlay | Fullscreen)

state: WindowState (Tiled | Floating | Fullscreen | MinimizedLike)

geom: Rect (last known from Sway)

manual_geom_override: bool (if user moved it)

cluster: ClusterId

transient_for: Option<WindowId> (dialogs)

tags: HashSet<String> (optional)

OutputState

name: String

geom: Rect (output bounds)

scale: f32

viewport: Viewport (if per-monitor)

focused: bool

Geometry pipeline: “intent → plan → apply”

Intent: user input or WM event triggers a desired state change

zoom changed, cluster moved, new window created, focus changed, output hotplug

Plan: compute target rectangles for affected windows

LayoutEngine::compute(canvas_state) -> HashMap<WindowId, TargetRect>

Apply: send minimal diff commands to Sway

only move/resize windows whose target differs beyond a threshold

This separation is the difference between “works sometimes” and “predictable.”

Deterministic layout rules (v1)

Keep it simple and consistent:

Overview

Don’t move real windows.

Your shell draws cluster cards (and maybe a list of windows per cluster).

Cluster zoom

Choose a bounding rect (work area minus panel)

Tile windows deterministically:

1 window: full

2 windows: split

3+: columns or BSP with stable ordering

Focus zoom

Focused window gets ~70–80% of area

Others become “strip” (vertical or horizontal) ordered by recency

Stable ordering is crucial (e.g., by last focus time). Users will hate “window shuffle.”

IPC layer and integration strategy
In Sway-prototype mode (recommended first)

You’ll use two IPC “planes”:

Read plane (events + state)

swayipc subscribe: window, workspace, output, binding, shutdown events

occasional get_tree, get_workspaces, get_outputs snapshots

Write plane (commands)

command: move/resize/focus, mark, criteria selectors

Practical Sway mechanics to lean on

Use marks to keep a stable handle to windows:

assign mark --add __contwm_<id> (or based on container id)

then target by [con_mark="..."] in commands

Maintain a mapping: WindowId <-> mark <-> cluster

Prefer batching: send a single command string with ; separated operations per “frame”

Example apply cycle (conceptually):

For each window needing change:

"[con_id=123] floating enable, resize set width height, move position x y"
(You’ll refine based on whether you use floating or tiled control.)

Important limitation: Sway tiling is tree-based; absolute positioning is easiest with floating windows. A workable hybrid:

In “Cluster” and “Focus” zoom, keep windows tiled and manipulate layout via splith/splitv + layout + resize set + move in container tree.

If that gets too fiddly early, prototype using floating for everything inside a dedicated workspace, then migrate to tiling control.

Clean internal IPC (inside your shell)

Even if everything is one binary today, define an internal protocol so you can split later:

contwmctl (client) → contwmd (daemon)

Transport: local Unix socket with JSON messages or D-Bus (zbus)

Messages:

SetZoom { level }

Pan { dx, dy }

CreateCluster { name, pos }

MoveWindowToCluster { window, cluster }

RenameCluster { ... }

GetState

ReloadConfig

This makes it easy to:

keep input handling in the panel

keep layout logic in a daemon

later swap Sway backend for wlroots backend without changing UI clients

Backend abstraction (so you don’t rewrite everything later)

Define a trait:

trait WmBackend {
  fn snapshot(&self) -> Snapshot;
  fn subscribe(&self) -> EventStream;
  fn apply(&self, plan: &Plan) -> Result<()>;
}

Phase 4 data contract

Coordinate space

Use world/canvas coordinates for cluster model data; treat screen pixels as a rendered projection only.

Units are logical canvas units (device-independent), with +X to the right and +Y downward.

Origin is canvas (0,0) at the initial overview center; viewport transforms map canvas coordinates to per-output screen space.

Required cluster-card fields (overview)

id: ClusterId (stable, unique)

position: Vec2 (canvas x,y center for hit-testing and draw placement)

size_or_radius: f32 or Vec2 (choose one canonical shape metric and keep it versioned)

title: String (user-visible cluster name)

window_count: u32 (number of windows currently assigned)

active: bool (cluster matches active_cluster)

focused: bool (cluster contains currently focused window)

Overview viewport model

center: Vec2 in canvas coordinates

scale: f32 (canvas units to screen mapping)

bounds: Rect in canvas coordinates for clamping/visibility logic

per_output_viewport: Option<HashMap<OutputName, Viewport>> for multi-monitor overrides; default to global viewport when absent.

Ownership and sync rules

Daemon state is source of truth for cluster entities, positions, viewport, and focus/active flags.

UI overlay owns only transient interaction state (hover, drag-in-progress, local animation/interpolation buffers).

UI sends intent messages (begin-drag/update/end-drag, pan, zoom) and must reconcile to daemon-acknowledged state on every state tick.

Persistence policy

Save immediately on durable model edits: cluster create/delete/rename, cluster assignment changes, and drag end (final position commit).

Save on debounce (e.g., 150–500 ms) for high-frequency viewport updates during pan/zoom.

Force flush pending debounced state on clean daemon exit/shutdown.

Conflict rules during drag

While dragging, keep a base_version/model_revision from drag start.

If external updates move the same cluster before drag end, apply deterministic resolution: last-writer-wins by revision/time and surface a brief UI indicator.

If external updates affect other clusters, merge them live without interrupting current drag target.

On drag end with stale base_version, either rebase delta onto latest position or reject with snap-back; pick one policy and keep it consistent.

Add “snap-to” and inertial feel later (optional)

Phase 4 Interaction Spec

State machine

OverviewIdle -> OverviewPanning when Mod+PointerMiddleDrag, Mod+PointerLeftDrag on empty canvas, or Mod+Arrow key repeat starts.

OverviewIdle -> OverviewSelecting when PointerLeftClick lands on a cluster card.

OverviewSelecting -> ClusterZoom when Enter is pressed (selected cluster) or PointerLeftDoubleClick hits the same cluster.

OverviewIdle/OverviewSelecting/OverviewPanning -> ClusterZoom when Mod+Enter is pressed and a cluster is selected.

Overview* -> PreviousZoom when Esc is pressed with no active drag and no modal UI open.

OverviewDraggingCluster -> OverviewIdle when PointerLeftRelease commits, Enter commits, or Esc cancels and restores pre-drag position.

Pan/zoom controls (overview)

Keyboard pan: Mod+Arrow pans by 96 px per keypress, 384 px with Shift.

Keyboard zoom: Mod+Plus and Mod+Minus adjust overview scale by 12% steps around viewport center.

Pointer pan: hold Mod and drag empty canvas with PointerLeft, or drag with PointerMiddle without modifiers.

Pointer zoom: Mod+WheelUp/Down (or Ctrl+Wheel on touchpads) zooms around pointer anchor.

Clamp zoom scale to [0.35, 2.50]; clamp viewport center after each pan/zoom to visible canvas bounds.

Overview interaction performance contract (v1 minimum)

Target frame behavior during drag/pan/zoom

Target 60 FPS interaction on typical desktop hardware (<=16.7 ms frame budget) for overview drag/pan/zoom when rendering up to 30 visible cluster cards.

Accept temporary degradation to 45 FPS (<=22.2 ms) during burst operations (output hotplug, first frame after zoom-level switch, or simultaneous cluster + viewport updates), but recovery to 60 FPS-equivalent pacing must occur within 500 ms.

If frame time exceeds 33 ms for more than 3 consecutive frames during active gesture, system must prioritize input + transform updates and defer non-critical visuals (drop shadows, secondary labels, expensive edge guides) until frame time recovers.

Implementation constraint: overview rendering path must keep cluster-card draw cost O(visible_cards) and avoid synchronous WM tree queries during gesture frames.

Throttling/debouncing rules for model writes during drag

During pointer drag, update overlay-local preview every coalesced input frame, but throttle daemon model position writes to at most 30 Hz (one write every ~33 ms).

Persisted state writes (disk/config snapshot) must be debounced to drag-end commit, with a safety flush at 250 ms if gesture stalls without release.

Viewport pan/zoom state writes may be sent at up to 20 Hz during active gesture and must always send a final unthrottled commit on gesture end.

Implementation constraint: overlay must maintain ephemeral drag state that can be rendered independently of daemon acknowledgment to prevent round-trip latency from blocking visual motion.

Render invalidation strategy for overlay

Default to dirty-region invalidation for overview interactions: invalidate previous and current cluster-card bounds (plus 8 px padding for shadows/guides), changed snap-guide segments, and HUD text rects.

Use full-overlay redraw only for structural changes (output geometry change, scale-step crossing that changes card LOD tier, theme/font swap, or debug overlay toggle).

Implementation constraint: renderer must track per-frame invalidation rectangles and union them; fallback full redraw is allowed only when dirty region union exceeds 60% of overlay area.

Event coalescing policy for high-frequency input

Pointer motion and wheel/scroll events are coalesced to one logical update per render tick; consume the most recent position/delta and accumulate wheel deltas since previous frame.

Input sampling target: 120 Hz max ingestion, 60 Hz presentation. If ingestion exceeds budget, drop intermediate motion events but never reorder press/release or direction changes.

Keyboard repeat pan events should coalesce by summing deltas within the same frame window.

Implementation constraint: event queue must timestamp raw events in monotonic time and expose both raw count and coalesced count for diagnostics.

Telemetry/logging hooks for jank diagnosis

Emit latency markers for each interaction frame: input_received_ts, coalesced_dispatch_ts, frame_build_start_ts, frame_submit_ts, and present_done_ts (when available).

Record derived metrics: input-to-dispatch, dispatch-to-submit, submit-to-present, and end-to-end input-to-present latency.

When end-to-end latency exceeds 50 ms for 3 consecutive frames, emit structured warn log with gesture type, visible card count, dirty region area %, and whether full redraw fallback was active.

Expose a lightweight ring-buffer trace endpoint (e.g., last 300 frames) via contwmctl dump-state --perf for post-hoc jank inspection without enabling verbose logging globally.

Selection behavior

Single click on cluster card: selects cluster, does not change zoom level, and updates focus ring.

Double click on selected cluster card: enters Cluster zoom for that cluster.

Enter: if cluster selected in Overview, dive into that cluster (same as double click).

Esc behavior priority: cancel drag preview -> close transient overlay panels -> return to previous zoom level.

If no selection exists, Enter is a no-op and should emit a lightweight hint in overlay status text.

Drag lifecycle

Drag start threshold: 8 px pointer movement from press point before card enters dragging state.

Live preview: card follows pointer in overlay space; connected edges/nearby snap guides render in preview layer only.

Snap preview: show target ghost position when within 24 px of grid line, neighboring card centerline, or output work-area center.

Commit keys: PointerLeftRelease or Enter commits cluster position and writes to in-memory model.

Cancel keys: Esc or PointerRightClick during drag cancels and restores original position.

Accessibility and keyboard-only fallback

Keyboard move mode: select cluster, press M to enter move mode, Arrow keys move by 32 px (Shift for 128 px).

In move mode, Tab cycles snap candidates; Space toggles snap on/off; Enter commits; Esc cancels.

Announce move-mode state, current coordinates, and snap target via accessibility status channel.

All drag outcomes (commit/cancel/snap) must be reachable without pointer input.

Focus handoff rules

Entering Overview: save currently focused window id, move input focus to overlay root, and set window focus freeze flag.

While in Overview, window tree focus must not change unless user explicitly dives into cluster/focus mode.

Leaving Overview via dive: hand focus to cluster's last_focus window, else first tiled window, else workspace container.

Leaving Overview via Esc/back: restore saved pre-overview window focus if it still exists, else fallback to active cluster last_focus.

On overlay dismissal, clear focus freeze flag after backend confirms focus command applied.

Multi-output and viewport edge handling

Maintain per-output overview viewport origin/scale; pan/zoom affects only focused output unless "link outputs" is enabled.

When pointer crosses output boundary during pan/drag, continue interaction in global canvas space but clamp previews to each output work area.

If selected cluster is outside current output bounds after zoom, animate viewport recenter to nearest visible point on that output.

Never place cluster card commit position outside global canvas min/max bounds; if exceeded, project to nearest valid point.

On output hotplug/remove during Overview, remap viewport to primary output and preserve relative cluster coordinates.

Phase 4 exit criteria (concrete checks)

1. Cluster creation + spatial persistence
   - User can create a new cluster from Overview, position it on the canvas, and drag/reposition any existing cluster.
   - Final cluster positions are persisted on commit and restored after daemon/app restart with no manual recovery steps.

2. Discoverable dive path with bounded interaction count
   - From Overview, user can select any currently visible cluster and enter Cluster/Focus mode in at most 2 interactions:
     - Path A: single click select + Enter.
     - Path B: double click to dive.
   - If selection is missing, UI must expose a hint and keep state unchanged (no accidental mode switch).

3. Pan/zoom selection stability
   - During overview pan/zoom, currently selected cluster remains selected until explicit selection change or mode exit.
   - If selected cluster moves partially or fully off-screen, viewport logic must provide recenter/discoverability behavior (per spec) without dropping selection state.

4. Keyboard-only parity
   - A keyboard-only path exists for selecting, moving, and activating a cluster:
     - Select: focus traversal/selection keys in Overview.
     - Move: enter move mode (`M`), adjust with Arrow/Shift+Arrow, commit with Enter, cancel with Esc.
     - Activate/dive: Enter from selected cluster.

5. Phase 2/3 regression guardrails
   - Repeated enter/exit cycles between Overview <-> Cluster <-> Focus do not regress deterministic layout behavior from Phase 2/3.
   - Specifically, no new window shuffle/flicker, and previously stable cluster/focus layout outcomes remain stable across at least 20 consecutive transitions.

Manual verification checklist (mapped to criteria)

- [ ] C1: Create two test clusters in Overview, drag both to distinct positions, restart daemon/session, verify positions match pre-restart placements.
- [ ] C2: For every visible cluster in current viewport, verify dive in <=2 interactions via (click+Enter) and (double click).
- [ ] C3: Select a cluster, perform aggressive pan/zoom in/out and across output edges, verify selection is preserved and cluster can be rediscovered/recentered without reselecting.
- [ ] C4: Unplug pointer input (or do not use mouse), select cluster via keyboard, move it with `M` + arrows, commit/cancel flows, and dive with Enter.
- [ ] C5: Run 20 Overview <-> Cluster <-> Focus cycles across at least 3 clusters with mixed window counts (1, 2, 3+), confirm no Phase 2/3 layout regressions.

Phase 5 — Persistence + robustness (make it daily-drivable)

Persist:

clusters, positions, window assignment hints (by app_id + title patterns)

last active cluster + viewport

Robust special-cases:

fullscreen windows (treat as temporarily overriding zoom)

dialogs transient_for parent (keep near parent / don’t strip)

multi-output behavior (at least don’t break)

Exit criteria: restart session and your landscape comes back sensibly.

Phase 6 — Performance polish (before compositor work)

Reduce IPC chatter:

only apply diffs above thresholds

batch commands

avoid frequent get_tree (prefer event-driven state)

Optional: add simple animations in the UI overlay only (not moving windows smoothly yet)

Exit criteria: no “constant resizing” feeling; CPU stays sane.

Phase 7 — Decide if/when to roll compositor

Once the model is validated and people like the interaction:

Start wlroots compositor if you need:

smooth transitions

true scene graph integration (thumbnails, blur, perfect zoom)

input policy / gestures deeply integrated

The key is you’ll already have:

the state model

the layout engine

the internal IPC protocol

the UX proven

So the compositor becomes “new backend,” not “rewrite the product.”

A practical v1 implementation tip

Treat your Sway prototype as a behavioral simulator:

“Overview” is your UI overlay (not actual window thumbnails at first)

“Zoom” is discrete modes with deterministic layouts

“Pan” moves the conceptual viewport and changes active cluster, not physical window coordinates

That keeps it feasible now and still maps cleanly to a native compositor later.
